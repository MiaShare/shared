Architecture: how a call flows end-to-end

expand
1. PSTN → Twilio

Caller dials +14788007559. Twilio receives the call.

2. Twilio → your laptop

Twilio POSTs to /webhooks/voice/{voiceCallSid} at your tunnel URL.

3. Lumion routes

CallRoutingService → InitialStep → AiAgentBridgeStep returns <Dial><Sip> with x-lumion-call-ref header.

4. Twilio bridges to OpenAI

Twilio dials sip.api.openai.com over TLS. Realtime SIP picks up.

5. OpenAI → Lumion (accept)

OpenAI POSTs realtime.call.incoming to /webhooks/openai/realtime. HMAC verified. Lumion mints a VoiceCallToken, calls acceptCall, stamps openai_call_id.

6. Tools during the call

Each MCP tool call lands at /mcp/voice-agent (bearer token, sliding TTL). ask_brain dispatches to gpt-5 via CortexService.

Trust the tunnel host (avoid signature failures)

Tunnels rewrite Host, send X-Forwarded-Proto: https, and inject a non-Herd hostname. If App\Http\Middleware\TrustProxies trusts '*' you're fine. If it's narrowed, add the tunnel host. Symptom of misconfig: the OpenAI webhook signature middleware rejects every webhook because request()->fullUrl() reports http://.

OpenAI account — from scratch

TODO

You lost the original account, so you're rebuilding the OpenAI side end-to-end. Five things have to be true before a call can land: project, API key, billing, realtime webhook, model access (gpt-realtime + gpt-5).

4.0 Account, organization, billing
  1. Sign up / sign in at platform.openai.com. Use a Lumion email so you don't lock yourself out again.
  2. If you're creating a fresh org: Settings → Organization → General. Set name and invite teammates so the next person who has to test this isn't blocked the way you are.
  3. Settings → Billing → add a payment method and put $20+ of credits on the org. Realtime + gpt-5 are paid-only — no credits = silent 401-style failures and the agent never speaks.
  4. Note the org ID (top-right org switcher → copy ID — looks like org-xxxx). Lumion doesn't read this directly, but you'll want it when filing access requests.
4.1 Create the Lumion Voice project
  1. Settings → Projects → Create project. Name it something clear like lumion-voice-{your-name}-dev.
  2. Open the new project and copy the project ID from the project switcher / project settings page — format proj_….
  3. Drop it into .env:
    .env
    OPENAI_REALTIME_PROJECT_ID=proj_YOUR_NEW_ID

Why this matters at runtime: Lumion builds the SIP URI as sip:{projectId}@sip.api.openai.com;transport=tls?x-lumion-call-ref=…. OpenAI uses the user-part of that URI to route the SIP call to your project's webhook. Wrong project id = SIP INVITE goes nowhere. (See AiAgentBridgeStep.php:69.)

4.2 Create a service-account API key
  1. Inside the new project: API keys → Service accounts → Create service account. Service-account keys are tied to the project, not your user — exactly what we want for headless use, and they survive your personal access changing.
  2. Name it lumion-voice-runtime.
  3. When the secret appears (it's shown once), copy it (format sk-svcacct-…) and replace the existing line in your .env:
    .env (replace existing line)
    OPENAI_API_KEY=sk-svcacct-YOUR_NEW_KEY
  4. Sanity check from your terminal:
    terminal
    curl -sS https://api.openai.com/v1/models \
  -H "Authorization: Bearer $OPENAI_API_KEY" | head -40

    Should return a JSON list of models. 401 = bad key. 403 = wrong project / billing not set up.

4.3 Confirm model access (gpt-realtime + gpt-5)

Two models, two reasons:

gpt-realtime-1.5 The voice agent itself — handles the conversational surface over SIP. Configured per-profile in ai_voice_agent_profiles.model; default is set in config/openai-realtime.php.
gpt-5 The "brain" the realtime agent consults via ask_brain for non-trivial reasoning. Pinned by prompts.id = 14's model column. Without gpt-5 access, AskBrainTool falls back to a degraded canned response and the agent gets dumb.
  1. Check availability in the project: platform.openai.com/docs/models — log in, switch to your new project, and search for gpt-realtime and gpt-5. If a model isn't listed, your org's tier doesn't have access yet.
  2. Or hit the API:
    terminal
    curl -sS https://api.openai.com/v1/models \
  -H "Authorization: Bearer $OPENAI_API_KEY" \
  | grep -E '"id":\s*"(gpt-realtime|gpt-5)'
  3. If gpt-5 isn't available, Settings → Limits shows your tier; load more credits to climb tiers, or open a request via the platform's "Request access" UI for the model.
  4. Realtime SIP is generally available on paid orgs. If the inbound webhook never fires after Step 4.4, fall back to the gpt-realtime-1.5 model name — older Realtime model names (gpt-4o-realtime-preview) require a different model string in config/openai-realtime.php default and per-profile model column.
4.4 Configure the realtime webhook

OpenAI needs to know where to POST realtime.call.incoming / .ended / .failed. Step 3 must be live first — you need your tunnel URL.

  1. Open Project → Webhooks in the new project. (If that exact path 404s, look under Project Settings → Integrations → Webhooks; OpenAI's nav has shifted.)
  2. Create endpoint:
    • URL: https://<your-tunnel-host>/webhooks/openai/realtime
    • Description: lumion-voice-dev-{your-name}
    • Events to send: realtime.call.incoming, realtime.call.ended, realtime.call.failed
  3. OpenAI shows the signing secret once on creation — format whsec_…. Drop it into .env:
    .env
    OPENAI_REALTIME_WEBHOOK_SECRET=whsec_YOUR_NEW_SECRET
  4. Reload Laravel config so the new key is picked up:
    terminal
    php artisan config:clear
  5. Test the webhook before dialing. OpenAI's webhook UI usually has a "Send test event" button — fire one and watch tail -f storage/logs/laravel.log on your laptop. You should see the request hit, signature middleware pass, and the controller log "unknown event type" (test events aren't realtime.*). That's success — round trip works.
Symptoms of webhook misconfig

401 / "invalid signature": wrong secret OR TrustProxies doesn't trust the tunnel (so request()->fullUrl() returns http:// and HMAC computes against the wrong canonical URL).

404: tunnel host wrong, or you typed an apex+subdomain like https://acme.<tunnel>/.... The route is on the apex.

Times out / no log entry: tunnel is dead or pointing at the wrong Herd site.

⚠

Things that bite

OpenAI webhook returns 401 / signature invalid

Cause: wrong OPENAI_REALTIME_WEBHOOK_SECRET for this URL. Fix: re-copy the secret from the OpenAI dashboard webhook bound to your tunnel.

Signature rejects despite correct secret

Cause: App\Http\Middleware\TrustProxies doesn't trust the tunnel; request()->fullUrl() returns http://... instead of https://.... Fix: trust '*' for proxies, or add the tunnel host explicitly.

OpenAI accepts call, then nothing — agent silent

Cause: MCP server URL unreachable; OpenAI can't fetch tools. Fix: curl https://<tunnel>/mcp/voice-agent from another network — must respond. Verify OPENAI_REALTIME_MCP_SERVER_URL matches your live tunnel.

Compliance exception, call short-circuits before SIP bridge

Cause: contact opted out / quiet hours / DNC / institution suspended. Fix: tinker the contact: optedOut === false, check voice_communication_preference, time of day.

AskBrainTool returns degraded fallback

Cause: voice-agent-ask-brain prompt missing or inactive. Fix: SELECT id, identifier, is_active FROM prompts WHERE identifier="voice-agent-ask-brain"; — should be id 14, active 1. If missing, re-run the seeder.

Call enters menu fine, AI profile picked, but <Dial><Sip> times out at Twilio

Cause: OPENAI_REALTIME_PROJECT_ID wrong, or service key has no realtime scope. Fix: check OpenAI dashboard logs for the project; ensure the API key is allow-listed for Realtime.

Long call cuts off at ~35 minutes

Cause: absolute token TTL hit. Fix: override OPENAI_REALTIME_TOKEN_ABSOLUTE_TTL_MINUTES for a long test, but reset it for prod.

php artisan ... exits 255 with no output

Cause: Telescope tries to write a terminate-time log row against tables that just churned (e.g. shortly after a db:wipe + reseed). Fix: either set TELESCOPE_ENABLED=false in .env while testing, or run php artisan migrate --path=vendor/laravel/telescope/database/migrations to repair telescope. For ad-hoc DB inspection, raw mysql against the forge database (the verification queries above) is fine and bypasses the issue entirely.
