CRTX-604 — Fix AI Response Truncation

VERIFIED March 28, 2026 • Branch: lhilton/crtx-604-ai-truncation

Problem Summary

1 — Problem

The AI Assistant's responses were being truncated mid-sentence during complex, multi-tool analytical outputs. Users received incomplete analyses that ended abruptly — mid-word or mid-number — making it obvious something failed. This was a distinct failure mode from silent failures (CRTX-602): the AI started generating a response but the output was cut off before finishing.

Root Cause

Prism defaults max_output_tokens to 2048 (defined in vendor/echolabsdev/prism/src/Concerns/ConfiguresModels.php:9). The AgentOrchestrator never called ->withMaxTokens() to override this default. Additionally, processChunk() ignored $chunk->finishReason, so truncation was completely invisible to both the system and the user. ROOT CAUSE

Solution Summary

2 — Solution

Two-part fix: (1) raise the token ceiling 8× and make it configurable, and (2) detect when truncation still occurs and notify the user instead of silently cutting off. 6 FILES CHANGED

Before (Broken)
max_output_tokens: 2048 (Prism default, not configurable)
Truncation detection: None — finishReason ignored
User notification: None — response silently cut off
Summary service: 2048 default, truncated summaries cached as valid
After (Fixed)
max_output_tokens: 16384 via config('ai.max_output_tokens')
Truncation detection: Tracks FinishReason::Length per chunk
User notification: Ephemeral notice streamed (not persisted to DB)
Summary service: 4096 limit, truncated summaries blocked with sentinel

AgentOrchestrator Changes

$stream = Prism::text() ->using('openai', $model->value) ->withSystemPrompt($systemPrompt) ->withMessages($messages) ->withTools($tools) ->withMaxSteps($maxSteps)+ ->withMaxTokens($maxOutputTokens) ->asStream();
// Post-persist ephemeral notice+ if ($finishReason === FinishReason::Length && $accumulatedText !== '') {+ $onTextDelta("\n\n---\n*This response was truncated due to length...");+ }

ContextSummaryService Changes

$result = Prism::text() ->using('openai', $prompt->model ?? AiModel::GPT_5_NANO->value) ->withSystemPrompt($prompt->system_prompt ?? '') ->withMessages([new UserMessage($userPrompt)])+ ->withMaxTokens(4096) ->generate();- return $result->text;+ if ($result->finishReason === FinishReason::Stop && $result->text !== '') {+ return $result->text;+ }+ // Truncated or empty → return sentinel to prevent retry loop+ return '';

Key design decision: The truncation notice is ephemeral — it is streamed after the message is already persisted to the database. This means it appears in the active session but disappears on reload. This prevents the truncation hint from polluting conversation context on subsequent AI turns.

Why This Fix Resolves the TripleTen Issue

FactorBeforeAfter
Token ceiling 2048 tokens (~500 words) 16384 tokens (~4000 words)
Truncation detection None — finishReason ignored FinishReason::Length tracked per chunk
User notification Silent cutoff — looks broken Ephemeral "truncated" notice + "ask me to continue"
Summary caching Truncated summaries cached as valid Sentinel prevents caching broken text
Retry loop Potential infinite LLM calls for summaries Sentinel stops retries at same msg count
Configurability Hardcoded Prism default config('ai.max_output_tokens') — adjustable per environment

The TripleTen conversation had complex analytical responses that regularly exceeded 500 words. With 2048 tokens, every response approaching ~500 words was silently cut off. The 16384 ceiling provides ~4000 words of headroom, which is sufficient for even the most complex multi-tool analytical outputs. If truncation still occurs at this ceiling, the user now sees a clear notice and can ask the AI to continue.
