TL;DR

InboundReplyAgent (app/Domains/CustomerJourney/Services/InboundReplyAgent.php) is a structured-output SMS/email agent with a rich context payload, a strict prompt, and a fan-out of post-dispatch side-effects in DispatchInboundReply. The Vapi inbound voice agent (BuildVapiAssistantConfig.php) is a realtime LLM with function tools and a deliberately lean system prompt.

The two systems are shaped differently — InboundReplyAgent expresses capabilities as JSON output fields + post-dispatch behaviors, Vapi expresses them as function tools + assistant context + prompt rules. So the gap list cannot just be "tools missing". It splits into four buckets:

  • A. Function tools — 6 net-new gaps + 2 already-scoped.
  • B. Context loaders — 5 confirmed gaps. Vapi inbound currently runs with significantly less grounding context than InboundReplyAgent.
  • C. Post-call signals — 4 gaps.
  • D. Prompt rules — 5 gaps, 1 already partially present.

The single highest-leverage gap is Bucket B: the realtime voice agent does not see candidate journey plays, recent outreach events, conversation state, the funding summary, or any inbound-call session context analog — which means it cannot accurately route callers responding to recent SMS/email outreach. Two function-tool gaps (send_link, create_task_for_human) are already scoped under PRAX-1159 / PRAX-1166 and are listed here only for completeness.

Bucket B

Recommended build order (Bilal's input requested)

A defensible order based on dependencies + ROI, not authoritative — Bilal triages:

  1. Bucket B as one workstream (BuildInboundCallContext combining candidate_plays + recent_automation_events + conversation_state + funding_summary headline + inbound session context). Highest ROI, unblocks future buckets. ~2 weeks.
  2. Bucket D rows 1, 3, 4 (anti-fabrication tighten + loop-breaker + recommendation hand-off). Cheap, high quality-of-output payoff. ~1 week.
  3. Bucket A schedule_callback — most-requested mid-call action per ticket signal. ~1 week.
  4. Bucket A update_contact_name + mark_asked_for_name (same migration area). ~1 week.
  5. Bucket C row 3 (post-call re-evaluation job). ~3 days.
  6. Bucket C row 2 + row 1 combined (post-call play attribution + transcript name backstop). ~1.5 weeks.
  7. Bucket A lookup_recent_conversation + lookup_recent_outreach — only if Bucket B's lite versions aren't enough. May not be needed. ~1 week.
  8. Bucket D row 5 (opt-out handling) — pending product decision.
  9. Bucket D row 2 (branch routing) — defer; not obviously needed.

Total if all built: ~7-9 weeks single-dev, much less if parallelized.

Appendix A — InboundReplyAgent JSON output schema

{
  branch: "active_play" | "no_active_play" | "new_contact",
  channel: "sms" | "email" | "voice",
  reply_body: string,
  reply_subject: string | null,           // email only
  chosen_play_id: int | null,             // active_play branch only
  chosen_journey_id: int | null,
  chosen_journey_stage_id: int | null,
  rationale: string,                      // 1-2 sentences, audit log
  commits_to_call: boolean,
  call_at: ISO8601 | null,
  call_brief: string | null,              // required when commits_to_call=true
  requests_human_followup: boolean,
  human_followup_summary: string | null,  // required when requests_human_followup=true
  contact_name_update: {
    first_name: string | null,
    last_name: string | null,
    rationale: string,
    is_correction: boolean
  } | null,
  asked_for_name: boolean                 // soft name-ask in this turn
}

Appendix B — Parked / out of scope

  • PlayAutomationGeneratorAgent — separate agent that generates automation sequences; not part of inbound reply path.
  • PlayAutomationActionEventReevaluatorAgent — referenced by Bucket C row 3 as the job target, not a tool to port.
  • TransitionAgentEvaluator — journey transitions; downstream of inbound replies, not a peer capability.
  • InboundReplyAgent test-mode (synthetic contact, debug transcript) — admin-only feature for tuning prompts; no production analog needed in Vapi.
  • InboundReplyAgent data_dictionary — JSON field definitions for the LLM; only useful when the prompt carries the structured context. If Bucket B ships, port this as a sub-task.
