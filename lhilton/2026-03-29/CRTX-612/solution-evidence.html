Root Cause: applicant_id = NULL

The PaymentScheduleTool MCP tool loaded payment methods via $applicant->paymentMethods (Eloquent hasMany on applicant_id). However, 17% of all payment methods have applicant_id = NULL and are linked only through user_id. Both of Alana's payment methods fall into this category.

Solution: Use PaymentBroker for Payment Method Lookup

Fix

Fix: Replaced the applicant-based payment method lookup with PaymentBroker::getPaymentMethodOptionsForUser($user, $institution) — the canonical pattern already used by the student dashboard and payment capture flows. This queries by user_id (finds methods regardless of applicant_id), filters by institution's active payment drivers, and applies notDisabled() + customerSelectable() scopes. 2 FILES, +61/-59 LINES

Code Change

private function getPaymentMethod(...) {- $newestMethod = $applicants- ->map(fn (Applicant $applicant) => $applicant->paymentMethods- ->sortByDesc('created_at')->first())- ->filter()->sortByDesc('created_at')->first();+ $methods = app(PaymentBrokerInterface::class)+ ->getPaymentMethodOptionsForUser($user, $institution);+ $bestMethod = $methods->sortByDesc('is_default')->first();

Why this works: PaymentBroker::getPaymentMethodOptionsForUser() queries payment_methods.user_id instead of applicant_id, correctly finding the 17% of payment methods that have applicant_id = NULL. It also filters by institution payment drivers (preventing cross-institution leakage) and prefers the is_default method.
