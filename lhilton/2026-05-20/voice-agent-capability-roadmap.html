TL;DR

Port the InboundReplyAgent capability surface to the Vapi realtime voice agent in four sequenced phases. Phase 1 is the highest-leverage work — a single combined BuildInboundCallContext service that gives the realtime agent the journey context InboundReplyAgent already has (candidate plays, recent outreach, conversation state, funding summary headline). Without this, every later phase reads weaker context and produces weaker results.

Phases 2-4 layer on the matching function tools, post-call pipeline, and prompt-rule tightening. Two sibling tickets (PRAX-1159 send_link, PRAX-1166 create_task_for_human) already cover two related tools and are excluded from this roadmap. One product decision (opt-out caller handling) is needed before Phase 2 can finalize.

Why four phases (and what they unlock)

PhaseThemeWhy it goes here
Phase 1 Context + prompt foundation Every later phase depends on the agent having journey context + tight grounding rules. Building tools first against a context-blind agent produces tools that the LLM under-uses or misroutes.
Phase 2 High-value mid-call tools Once the agent has context, the two highest-value mid-call actions (schedule a callback, capture the caller's name) unlock real-world workflow parity with SMS/email outreach.
Phase 3 Post-call pipeline Keep downstream automation cadence consistent with what the call covered. Cheap once we agree to listen for Vapi's end-of-call-report.
Phase 4 Defensive backstops + optional depth Only pursued if Phase 1 context isn't rich enough on its own, or if Phase 2 tool fires miss real-world cases.

Context + prompt foundation

Give the realtime agent the journey grounding InboundReplyAgent already has, and tighten the prompt rules that govern how it speaks.
~3 weeks
parallel-build friendly
Build the BuildInboundCallContext service Context ~2 weeks

One combined service called from BuildVapiAssistantConfig at call start. Returns a single context payload bundling everything the realtime agent needs to know about the contact before the first turn.

Payload sections:

  • candidate_plays (lite) — caller's active journey plays via PlayContextService::getContextLite(). Always-on; the agent cannot pick a play it cannot see.
  • recent_automation_events (top 3) — last 3 outreach actions sent to the contact. Drives "you replied to my text about X" recognition.
  • conversation_state.current_program_interest — carry-forward so the agent doesn't re-ask which program the caller is interested in. Verify column persists on Contact (not session-scoped only).
  • funding_summary headline — 2-3 line summary baked into the prompt ("This school offers N programs; ask get_program_details for specifics"). Keeps the existing function tools as the primary surface but gives the agent enough to know what to offer.
  • inbound session context — analog of outbound's AutomationVoiceCallContext. Surface "we sent SMS 2hrs ago about deposit deadline" type framing. Likely just a curated view over the three sections above.
Maps from: InboundReplyAgent:L307 (voice context build), PlayContextService::getContextLite(), AutomationVoiceCallContext. Lands in BuildVapiAssistantConfig.php.
Tighten RenderSessionInstructions prompt rules Prompt ~1 week

Three additive rule edits, all direct copies / tightenings of what InboundReplyAgent's voice guidelines already enforce:

  1. Anti-fabrication tightening. The existing line ("Do not improvise facts outside that returned answer", RenderSessionInstructions.php:L95) covers tool output but not the intro/outro phrasing the agent composes around it. Add a CLAIM-LEVEL SELF-CHECK rule matching InboundReplyAgent: before stating any specific number, date, program name, or instructor name, locate the exact string in tool output / loaded context.
  2. Loop-breaker. If the agent's immediately preceding turn asked a clarifier and the caller didn't answer, do not re-ask — answer with what is known or defer to create_task_for_human. Source: InboundReplyAgent.php:L939 voiceReplyGuidelines().
  3. Recommendation hand-off. Once a program recommendation lands in a call, subsequent same-recommendation turns acknowledge briefly and return the floor to the caller without re-announcing. Source: same as above.
Lands in app/Domains/Communications/Services/Voice/AiAgent/RenderSessionInstructions.php.
2

High-value mid-call tools

Two new function tools matching the highest-impact JSON output fields from InboundReplyAgent.
~2 weeks
follows Phase 1 context
schedule_callback Function tool M ~1 week

Mid-call tool: the agent commits to an outbound callback at a concrete future time. Creates the same Automation + AutomationAction(Voice, scheduled) + AutomationActionEvent chain that DispatchInboundReply creates from InboundReplyAgent's commits_to_call=true output.

Tool inputs: callback_at (ISO 8601), call_brief (2-4 sentence brief). Tool returns voice-formatted confirmation ("Got it, we'll call you back tomorrow at 2 PM").

Maps from: DispatchInboundReply.php:L141 scheduleOutboundCall(). New action class + VapiScheduleCallbackToolController + route in routes/webhooks.php.
update_contact_name + mark_asked_for_name Function tool (pair) S ~1 week combined

Two tools that touch the same Contact column area — build together with one migration audit.

  • update_contact_name: caller corrects or provides their name mid-call. Inputs: first_name, last_name, is_correction (false = fill blanks only, true = overwrite). Reuses the same correction-vs-fill logic as ApplyInboundContactNameUpdate:L36. Implicitly sets the asked-for-name flag (the agent already got the name, no need to ask again).
  • mark_asked_for_name: agent has just done a soft name-ask in this turn. Sets the flag so consecutive turns in the same call (and the next call) don't re-ask.
Maps from: InboundReplyAgent contact_name_update + asked_for_name JSON fields. New action classes + 2 controllers + routes. Verify Contact column name for the asked-for-name flag during build.

Post-call pipeline

Listen for Vapi's end-of-call-report and run the same downstream side-effects InboundReplyAgent triggers.
~4 days
cheap, high consistency payoff
Post-call re-evaluation job Post-call S ~3 days

Vapi end-of-call-report handler kicks a ReevaluateContactAfterInboundReplyJob analog that re-runs PlayAutomationActionEventReevaluatorAgent. Keeps pending SMS/email outreach in line with what the call just covered (no double-texting "are you still interested?" 10 minutes after the caller said yes on a call).

Maps from: DispatchInboundReply.php:L173-179. Lands as a new job + Vapi server-message handler. Idempotency matters — guard against duplicate end-of-call-reports.
Inactive-journey suppression Post-call S ~1 day extension of above

Verify the reevaluator above honors journey.is_active=false (audit trail still writes, but new outbound is suppressed). If it does, this row is satisfied implicitly by shipping the row above. If not, add the gate.

Maps from: DispatchInboundReply.php:L108 + L115 + L136. Verify behavior, add the gate if missing.
4

Defensive backstops + optional depth

Only pursue if Phase 1 context proves insufficient or Phase 2 tools miss real-world cases. Hold these unless we see the gap in practice.
~2 weeks if all built
conditional
Transcript-mined contact name backstop Post-call M ~3-5 days

Post-call LLM pass over the transcript catches name updates the mid-call update_contact_name tool missed (caller says it casually mid-sentence, agent doesn't recognize as a name correction). Only worth building if Phase 2 update_contact_name ships and we observe missed updates in production.

Maps from: ApplyInboundContactNameUpdate:L36.
lookup_recent_conversation + lookup_recent_outreach Function tools M ~1 week

On-demand deeper lookup beyond the 3-message / 3-event summary Phase 1 bakes into context. Only worth building if Phase 1's headline counts prove insufficient.

  • lookup_recent_conversation — return more SMS/email/voice history when the agent needs to recall a specific past exchange.
  • lookup_recent_outreach — return more AutomationActionEvent rows for the same reason.
Maps from: contact_context.messages + recent_automation_events in InboundReplyAgent. Hold until Phase 1 ships and call recordings show the agent fishing for more context than it has.

Open product decisions needed before Phase 2

Opt-out caller handling — what does Vapi do when a contact with ai_outreach_enabled=false calls in? Decision

DispatchInboundReply:L84 creates a Task instead of an AI reply when the contact has opted out. The Vapi voice path does not currently read this column (zero hits for ai_outreach_enabled under app/Domains/Communications/Services/Voice/).

Options:

  1. Answer normally — opt-out means "don't us contact them". Inbound is caller-initiated, so the opt-out doesn't apply. (Recommendation.)
  2. Answer with a short scripted "your account is set to no AI; I'll have a teammate call you back" + create a Task.
  3. Hang up + create Task.
  4. Route to voicemail.

Default recommendation lets Phase 2 ship without a product decision blocking it. Worth confirming explicitly with Bilal before locking.

Deferred / declined

Attribute the call to a specific journey play/stage (mid-call or post-call)
Out of scope per direction. Both the mid-call log_play_attribution tool and its post-call transcript-classification alternative are dropped from the roadmap. JourneyActivityLog entries from Vapi calls will not be play-attributed for now.
Three-branch routing (active_play / no_active_play / new_contact)
InboundReplyAgent uses explicit branch routing in a single LLM turn. Voice intent surfaces more dynamically across multiple turns — the caller states their context. Hold unless Phase 1 + Phase 2 in production show the agent confusing first-touch callers with active-play returners.
