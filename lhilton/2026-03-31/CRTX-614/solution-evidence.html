User Unable to Click Scheduling Links in AI Chat Widget

A prospective student interacting with the lead capture AI chat widget tried to schedule a campus visit. The AI provided scheduling links, but they appeared as partially-clickable text — the user repeatedly asked "where do I click?" across 31 messages, indicating significant frustration.

Root cause: UrlService::addPlainDomainLinks() used a regex with \b word boundaries that stopped matching at / characters. For a bare URL like calendly.com/example-school, only the domain calendly.com was linked. The path /example-school was left as plain text, producing a broken link that navigated to the wrong URL.

<!-- What the user saw (BEFORE fix) --> <a href="https://calendly.com">calendly.com</a>/example-school <!-- Only "calendly.com" was clickable. "/example-school" was plain text. --> <!-- Clicking the link went to calendly.com (homepage), not the scheduling page. -->
3 Solution Summary
Fix 1 — Primary

Regex Update: Capture Path Components

Updated addPlainDomainLinks() regex to capture optional path segments after bare domains. The callback now separates domain (lowercased) from path (case-preserved per RFC 3986). Ampersands are excluded from the path character class to prevent HTML entity absorption.

// UrlService.php - addPlainDomainLinks() $domainPattern = '/\b(?<![@\/])(...)+)\b(?![@])/i'; $domainPattern = '/\b(?<![@\/])(...)+)(\/[^\s<>"&]*[^\s<>"&.,;:!?\)\]\}])?(?![a-zA-Z0-9@])/i'; // Added optional path capture group after domain // Domain lowercased, path case preserved
Fix 2 — Defense-in-Depth

AI Prompt: Always Include https://

Added a URL formatting instruction to the lead capture agent system prompt: "Always include the full URL with https:// when sharing a link." This ensures GFM autolinks also catch URLs (belt + suspenders).

// CortexPromptMigrationSeeder.php // RULES FOR SHARING ACTIONS section: - Always include the full URL with https:// when sharing a link. For example: https://calendly.com/school, not just calendly.com/school.
