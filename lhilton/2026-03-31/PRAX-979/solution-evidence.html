PRAX-979: AI Enrollment Form Builder Intermittent Failures

Lumion AI Assistant — Fix Verified via Unit Tests & Database Analysis
Problem Summary

The AI enrollment form builder was failing intermittently (~67% failure rate) with "An unexpected error occurred while generating the form". Users had to retry multiple times for basic operations like renaming a page or creating a standard form. The AI chat would attempt the operation, report failure, and suggest contacting support.

Solution Summary

Root cause: a parameter mapping bug in PrismDispatcher.php that silently ignored the configured max_tokens=32768 for GPT-5 models, causing OpenAI to receive only the Prism framework default of max_output_tokens=2048. With only 2048 tokens for both reasoning and output, GPT-5.2 intermittently exhausted the budget.

BUG 1 GPT-5 branch skips withMaxTokens()

PrismDispatcher set $providerOptions['max_completion_tokens'] for GPT-5 models, but Prism's OpenAI handler never reads that key. It only uses $request->maxTokens(), which defaults to 2048.

// BEFORE (buggy) — PrismDispatcher.php lines 77-84 if (isset($options['max_tokens'])) { if ($isGpt5) { $providerOptions['max_completion_tokens'] = $options['max_tokens']; // SILENTLY IGNORED } else { $request = $request->withMaxTokens($options['max_tokens']); } }
// AFTER (fixed) if (isset($options['max_tokens'])) { $request = $request->withMaxTokens($options['max_tokens']); }

BUG 2 reasoning_effort silently ignored

PrismDispatcher set providerOptions['reasoning_effort'] but Prism reads providerOptions('reasoning') expecting { effort: "medium" }. Both streaming and non-streaming paths were affected.

// BEFORE $providerOptions['reasoning_effort'] = $options['reasoning_effort']; // AFTER $providerOptions['reasoning'] = ['effort' => $options['reasoning_effort']];

FIX 3 Better error messages & compact JSON

Replaced generic error message with actual error detail. Removed JSON_PRETTY_PRINT from form context serialization to reduce ~30% token overhead.
