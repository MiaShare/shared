What's new since v1

Six tools v1 did not cover.

send_link, school_local_time, plus four MCP tools: end_call, transfer_call, initiate_authenticated_callback, opt_out_of_voice. Single-call scripts in v2 hit every one.

update_contact_name picked up two new clear flags (PRAX-1175 / LUM-3119).

clear_first_name / clear_last_name let the model drop a stale slot when the caller shortens their name (e.g. CALLER CONTEXT shows "Bob Smith", caller says "actually my name is just Bob" → first_name="Bob", clear_last_name=true, is_correction=true). Covered in deep-dive 8.

lookup_recent_outreach picked up a task channel.

channel now accepts task for internal follow-up notes alongside sms, email, voice, all. Channel filter is system-prompt-enforced — mis-filtering is loud and observable in tool.invoked payload.

mark_asked_for_name gate is broader.

Now fires when EITHER first or last name slot is blank (PRAX-1177). Three reachable shapes: NAME UNKNOWN, FIRST NAME ONLY ON FILE, LAST NAME ONLY ON FILE. Deep-dive 7 covers all three.

Reply shape: result-only for substantive turns, request-complete only for suppressed turns.

v2 verifies barge-in works on every speaking tool. A tool.result row whose payload has both result AND a message: {type: "request-complete", content: "..."} block with non-space content is a bug. See barge-in verification.

Single-call script A — happy path, 14 tools

~7-8 minutes. Hits 11 direct tools plus end_call and transfer_call from MCP — 14 distinct tool invocations, two of them MCP. Call from the dev phone tied to contact 871 (existing history makes the lookup tools meaningful). The order is intentionally rearranged from v1 to put name-fix and KB asks LATER in the call so we can verify mark_asked_for_name persistence and idempotency caches.

Speak naturally with normal pauses.

Wait for the agent to finish each reply before delivering the next line. The stopSpeakingPlan VAD is tuned for natural speech and will barge over its own TTS if you step on it deliberately — that's a separate test (see barge-in).

Authorized-callback flow — 2 MCP tools gated behind callback

Two MCP tools only register on Authorized-scope sessions. The Authorized scope is minted by initiate_authenticated_callback after we've verified the caller's from_number matches a phone on the contact record. The callback session itself runs at a tighter token TTL (default 15 min absolute) than a normal session.

Prerequisite.

Call from contact 871's phone (+15129533841). The dispatch leg places an outbound call back to that same number with isCallback=true, bypassing opt-out and quiet-hours. The new call mints a fresh Authorized VoiceCallToken. The originating call is hung up immediately after dispatch.

Step 1 — Trigger the callback from a normal session

#SayExpect
1(wait for greeting on normal call)Greeting.
2
I need to update my account preferences but I want to verify it's me first.
Agent reads sensitive-action policy, then invokes initiate_authenticated_callback with a sanitized context_summary (10-500 chars, control chars stripped, ALL-CAPS headers neutralized to prevent prompt injection).
3(originating call hangs up: "Callback dispatched. The caller will receive an incoming call momentarily.")Twilio REST hangs up parent leg. Within seconds, your phone rings — that's the authenticated callback leg.

Step 2 — On the callback leg, exercise opt_out_of_voice

#SayExpect
1(answer callback)Greeting acknowledges this is a verified session.
2
Please stop calling me for marketing. Opt me out.
opt_out_of_voice fires (MCP tool, only registered on this Authorized scope). Writes ContactOptPreference row, CommunicationType::Phone marketing=false, updatedBy=$contact.
3
Thanks, that's all.
end_call fires.

Verification

-- Two calls: dispatch leg + authenticated callback leg
SELECT id, direction, from_number, to_number, ai_end_reason, created_at, ended_at
FROM cortex_forge.calls
WHERE contact_id = 871 AND created_at >= NOW() - INTERVAL 15 MINUTE
ORDER BY id;
-- Authorized token TTL is tighter (~15 min) than standard
SELECT id, call_id, scope, absolute_expires_at, idle_expires_at, created_at
FROM cortex_forge.voice_call_tokens
WHERE call_id IN (SELECT id FROM cortex_forge.calls WHERE contact_id = 871 AND created_at >= NOW() - INTERVAL 15 MINUTE)
ORDER BY id;
-- Opt-out row written from authenticated leg
SELECT id, contact_id, communication_type, marketing, updated_by_type, updated_by_id, created_at
FROM contact_opt_preferences
WHERE contact_id = 871 ORDER BY id DESC LIMIT 5;

Logging tips

After each test call, pull events with a single query so you can see tool order, payloads, and result text in one place. If a tool didn't fire when expected, the absence of its tool.invoked row is your signal — surrounding caller.transcript rows tell you what utterance the agent actually saw.

-- Full timeline for a call
SELECT occurred_at, type, tool_name, LEFT(text, 200) AS text_preview, payload
FROM cortex_forge.voice_call_debug_events
WHERE call_id = '<call-id>'
  AND type IN ('caller.transcript','agent.transcript','tool.invoked','tool.result')
ORDER BY occurred_at, id;
-- Which tools registered on this call (eligibility check)
SELECT DISTINCT tool_name FROM cortex_forge.voice_call_debug_events
WHERE call_id = '<call-id>' AND type = 'tool.invoked'
ORDER BY tool_name;
-- Tool idempotency check: same toolCallId fired twice?
-- (would indicate Vapi retry leaking past Cache::add idempotency key)
SELECT tool_name, JSON_EXTRACT(payload, '$.toolCallId') AS tool_call_id, COUNT(*)
FROM cortex_forge.voice_call_debug_events
WHERE call_id = '<call-id>' AND type = 'tool.invoked'
GROUP BY tool_name, tool_call_id
HAVING COUNT(*) > 1;
