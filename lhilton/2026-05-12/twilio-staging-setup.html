Twilio staging setup

End-to-end checklist that takes Lumion's staging environment (DigitalOcean Ubuntu droplet, Envoyer-deployed) to: "I can place a real call, send a real SMS, and pass a real Verify OTP from staging code, targeting https://lumion-cj-scenarios.cortex.staging.lumion.ai."

Steps

State persists in localStorage on this domain. Separate from the local-setup checklist.

Prerequisite: you've completed the local-setup guide. Staging reuses the same Twilio sub-account, the same Vapi org, and the same purchased phone number. Staging just takes over the webhook URLs from local. Step 14 covers switching back.

The whitelist gate from lhilton/twilio-phone-whitelist applies to staging too (any non-production env), so step 8 — adding your phone — is non-optional.

1. Prereqs

  • Envoyer access to the staging project for cortex.staging.lumion.ai. You'll edit env vars through the Environment tab.
  • SSH access to the staging droplet for php artisan commands. Confirm: ssh <user>@<host> 'echo ok'.
  • Branch deployed. Staging must be running a branch that includes the scenario auto-bind helper (ensureLocalTwilioNumberForInstitution in CreatesJourneyBase) added by lhilton/twilio-phone-whitelist. Without it, step 10 falls back to manual UI binding.
  • Horizon supervised on staging. Most setups run Horizon under Supervisor or systemd, restarted on deploy via Envoyer's hooks. Confirm with sudo supervisorctl status horizon (or equivalent).
  • Local guide complete. You need every SID + secret from twilio-local-setup steps 2–9.
Substitute throughout this guide: <user>@<host> = your droplet SSH credentials, <deploy-path> = the Envoyer project root on the droplet (typically /home/envoyer/<project> — check Envoyer → Project → Settings → Project Path).

Same Envoyer Environment tab. The staging admin host is your apex; webhooks land here regardless of which institution they belong to (route handlers look up the To number to resolve tenant).

TWILIO_VOICE_URL=https://admin.cortex.staging.lumion.ai

VAPI_MCP_SERVER_URL defaults to APP_URL + /mcp/voice-agent when unset. Setting it explicitly makes the configuration self-documenting and survives any future APP_URL change.

If Twilio webhooks 404 in the Console debugger (Console → Monitor → Logs → Errors), the apex Host context isn't accepting these routes on staging. Fall back to the institution subdomain — replace admin.cortex.staging.lumion.ai with lumion-cj-scenarios.cortex.staging.lumion.ai in every value above and re-Update. Tenant resolution under that Host is unambiguous.

7. Re-point Twilio + Vapi webhooks at staging

This step disconnects local. After it, every inbound SMS, every inbound voice call, every Vapi webhook lands on staging. Local devs who want to test will need to reclaim — see step 14.

Easy: voice:dev-sync from staging SSH

ssh <user>@<host>
cd <deploy-path>/current
php artisan voice:dev-sync \
    --tunnel=https://admin.cortex.staging.lumion.ai \
    --institution=lumion-cj-scenarios \
    --i-confirm-dev-vapi

Despite the command name, voice:dev-sync isn't tunnel-specific — --tunnel= is just "the public URL where webhooks should be sent." Pointing it at staging patches:

  • Twilio Phone Number → Voice URL + Voice Status Callback + Recording Status Callback
  • Vapi per-phone-number server config (URL + HMAC credential attachment)
  • The institution's TwilioNumber row in the DB

It does not patch Twilio's Messaging URL on the phone number, the Messaging Service inbound webhook, or the Vapi org-level Server URL — those are manual additions below.

Without --i-confirm-dev-vapi, the command pauses with an interactive "Are you using the Vapi DEV org?" prompt. Same as local — answer yes if so.

Manual additions (apply once after voice:dev-sync)

Console field Value
Twilio → Phone Numbers → your number → Messaging URL (POST)https://admin.cortex.staging.lumion.ai/incoming-sms-messages/twilio
Twilio → Messaging → your service → Integration → Inbound webhook (POST)https://admin.cortex.staging.lumion.ai/incoming-sms-messages/twilio
Vapi → Org Settings → Server URL (dashboard.vapi.ai/settings/organization)https://admin.cortex.staging.lumion.ai/webhooks/vapi (HMAC credential attached)

voice:dev-sync does not touch Vapi org-level settings. The org-level Server URL needs the same HMAC credential attached as the per-phone-number config; without it, org-level webhooks (assistant-request fallbacks) come in unsigned and the middleware 401s them.

8. Whitelist your phone (the staging gate)

Without this, every real SMS / Verify / Voice call from staging code throws Twilio sms blocked: recipient ending in …. The magic Twilio test number +15005550006 is auto-allowed by the boot whitelist; everything else needs a row.

  1. Open https://admin.cortex.staging.lumion.ai/admin and log in as a Lumion staff user.
  2. Sidebar → Developers → Twilio Allowed Recipients.
  3. Click Quick-add my phone (uses the phone column on your user record). If that column is null, fall back to New twilio allowed recipient with the free-form input — (415) 555-0123 normalizes to +14155550123 on save.

Cache (twilio.allowed_recipients, 24h TTL) refreshes write-through on save — no manual purge.

The scenario is idempotent — reuses the institution by its hard-coded lumion-cj-scenarios subdomain. Console output prints the seeded admin email (cj-admin-<timestamp>@test.com, password password). Because you set LOCAL_TWILIO_* in step 4, you'll also see:

That line confirms ensureLocalTwilioNumberForInstitution (in app/Scenarios/CustomerJourney/Concerns/CreatesJourneyBase.php) created or updated the SMS Channel + TwilioNumber row + paired VoiceSetting to point at your real number. 10. Verify the bind

Visit https://lumion-cj-scenarios.cortex.staging.lumion.ai/settings/phone-numbers and log in as the staging school admin from step 9. You should see a row with:

  • SMS number = LOCAL_TWILIO_NUMBER
  • Messaging Service SID = LOCAL_TWILIO_MESSAGING_SERVICE_SID
  • Phone Number SID = LOCAL_TWILIO_PHONE_NUMBER_SID

If the row is missing or the SIDs are wrong:

  • Branch on staging may not include the scenario helper — check that the deployed branch matches lhilton/twilio-phone-whitelist. Without the helper, you'll need to add the row manually here: SMS number = your E.164, Messaging Service SID = MGxxxx, Phone Number SID = PNxxxx.
  • LOCAL_TWILIO_* may not have made it into staging .env — re-check Envoyer + the SSH grep from step 4, then re-run the scenario.

11. Smoke test SMS

This step verifies the whitelist gate end-to-end, not real carrier delivery. The WhitelistGatedSmsService decorator sits in front of TwilioSms::send() via tagged container binding; we want to confirm the chain is wired and the whitelist passes.

$result = $service->send(new \Lumion\Shared\DTOs\SendSmsDto(

Optional: full pipeline via the staging UI

Open a Contact in lumion-cj-scenarios (staging) whose phone is your whitelisted E.164, click Send SMS, send. Same gate decorator regardless of caller.

Carrier delivery caveat

US carriers (T-Mobile, Verizon, AT&T) drop A2P SMS from long-code numbers without A2P 10DLC registration. Twilio still mints an SID and reports status: pending, then flips to undelivered with error_code: 30034 shortly after. This is org-level and not specific to staging vs local — same number, same carrier rules. See local guide step 17 ("Carrier delivery caveat") for the curl recipe to inspect message status by SID and the three resolution paths (Verify magic number / toll-free / A2P 10DLC). For dev smoke testing, use Verify (step 12) instead — +15005550006 always returns approved.

Trigger an OTP via any flow that uses TwilioSmsVerification (portal magic-link login, etc).

13. Smoke test Voice (Twilio + Vapi end-to-end)

Re-run if anything has drifted since step 7:

ssh <user>@<host>
cd <deploy-path>/current
php artisan voice:dev-sync \
    --tunnel=https://admin.cortex.staging.lumion.ai \
    --institution=lumion-cj-scenarios \
    --i-confirm-dev-vapi

Then either:

  • Inbound — dial your purchased Twilio number from any phone. Twilio bridges to Vapi, Vapi POSTs assistant-request to https://admin.cortex.staging.lumion.ai/webhooks/vapi, staging mints a per-call MCP token and replies with the assistant config, the conversational LLM uses MCP tools at https://admin.cortex.staging.lumion.ai/mcp/voice-agent. Watch the assistant-request handler: 
    tail -f <deploy-path>/current/storage/logs/laravel.log
  • Outbound — place a call from the staging in-app voice client (institution user with the right role can dial from a Contact). Same MCP wiring; this path uses POST https://api.vapi.ai/call directly.

If the call connects but goes silent after the greeting, check that VAPI_MCP_SERVER_URL in Envoyer matches https://admin.cortex.staging.lumion.ai. If /voice/twiml returns 502 in Twilio's debugger, see app/Console/Commands/Communications/VoiceDevSyncCommand.md "Common failure modes". If Vapi end-of-call webhooks 401, the HMAC credential isn't attached to the per-phone-number server config — re-run voice:dev-sync.

14. Switching back to local

Twilio + Vapi resources have one webhook destination at a time. While staging owns the webhooks, your local tunnel won't receive any inbound traffic. To reclaim for local development: 

# On your local machine, with the tunnel running
herd share --subdomain=<your-local-subdomain>

# In another terminal
cd /Users/leezilla/conductor/workspaces/miashare-app/tallahassee
php -d memory_limit=512M artisan voice:dev-sync \
    --tunnel=https://<your-local-subdomain>.sharedwithexpose.com \
    --institution=lumion-cj-scenarios \
    --i-confirm-dev-vapi

This re-points the same Twilio Phone Number + Vapi per-phone-number config back at your local tunnel. Staging stops receiving webhooks until someone re-runs the staging voice:dev-sync from step 7.

Field When local owns it When staging owns it
Twilio → Phone Number → Messaging URL<local tunnel>/incoming-sms-messages/twiliohttps://admin.cortex.staging.lumion.ai/incoming-sms-messages/twilio
Twilio → Messaging Service → Inbound webhooksame as abovesame as above
Vapi → Org Settings → Server URL<local tunnel>/webhooks/vapihttps://admin.cortex.staging.lumion.ai/webhooks/vapi
Outbound from local still partially works while staging owns the webhooks. A local tinker TwilioSms::send() will succeed as far as Twilio is concerned — Twilio's REST API is hit directly from your laptop, no webhooks involved. But any reply SMS, any delivery status callback, any inbound call routes to staging until you reclaim. Don't be confused by "outbound says success, inbound never arrives" — that's the resource-ownership split.
