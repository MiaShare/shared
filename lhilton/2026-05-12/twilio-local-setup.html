Twilio local-dev setup

End-to-end checklist that takes a fresh dev machine to: "I can place a real call, send a real SMS, and pass a real Verify OTP from local code, targeting the seeded lumion-cj-scenarios institution."

Steps

State persists in localStorage on this domain.

Why this guide exists. The branch lhilton/twilio-phone-whitelist adds a recipient whitelist gate. In any non-production env, SMS / Voice / Verify calls to numbers that aren't on twilio_allowed_recipients (or the magic test number +15005550006) are blocked by the WhitelistGated* decorators. So step 14 — adding your own number — is non-optional.

1. Prereqs

  • Herd is serving *.tallahassee.test (run herd open to confirm).
  • Branch checked out: git checkout lhilton/twilio-phone-whitelist.
  • DB up to date: php -d memory_limit=512M artisan migrate.
  • Horizon running in a separate terminal: php -d memory_limit=512M artisan horizon. Required for async voice jobs (transcription, Vapi cleanup) and any job-dispatched SMS path.

2. Twilio project

Pick a sandbox / dev Twilio project — not the production account. From the Console:

  • Account → API keys & tokens — capture Account SID + Auth Token.

Stash both in your password manager; you'll paste them into .env in step 10.

3. Twilio phone number

  • Phone Numbers → Manage → Buy a number — pick a US number with both Voice and SMS capability. A trial-account number is fine.
  • Capture the number's E.164 (e.g. +15558675309) and its Phone Number SID (PNxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx).
Trial-account caveat. Trial accounts can only send real SMS to numbers verified as caller-IDs. Fine for self-testing; to hit a teammate's phone, verify their number in the Console or upgrade. Twilio Verify works to any number — only raw outbound SMS has this restriction.

4. Twilio API key (for browser voice)

  • Account → API keys → Create API Key — type Standard.
  • Capture API Key SID (SKxxxx) and the one-time Secret. Lose the secret = create a new key.

Feeds TWILIO_API_KEY_SID / TWILIO_API_KEY_SECRET. Used by the browser-voice access-token endpoint (POST /voice/token). Skip if you only care about SMS / Verify.

5. Twilio Verify Services

  • Verify → Services → Create — make two services. Friendly names like "Lumion Local — SMS" and "Lumion Local — Email".
  • Capture each Verify Service SID (VAxxxx). One feeds TWILIO_SMS_VERIFICATION_SERVICE_SID, the other TWILIO_EMAIL_VERIFICATION_SERVICE_SID.

6. Twilio TwiML App

  • Voice → TwiML → TwiML Apps → Create. Voice URL + status callback can stay as placeholders for now — voice:dev-sync (step 19) overwrites them with your tunnel URL.
  • Capture the TwiML App SID (APxxxx). Required only if you intend to test browser-initiated voice.

7. Twilio Messaging Service

  • Messaging → Services → Create → "Notify my users".
  • In Sender Pool, add the phone number from step 3.
  • Capture the Messaging Service SID (MGxxxx).
A2P 10DLC. Brand + campaign registration is out of scope for local dev. Twilio Verify works without it. Raw outbound SMS to non-verified US numbers may filter at the carrier — to your own verified caller-ID it works on a trial account.

8. Tunnel (Herd Share with stable subdomain)

Herd Share wraps Expose — it spins up an HTTPS tunnel to your local Herd site and handles Host-header rewriting automatically, so Twilio's webhooks reach Laravel without bouncing off the local HTTP→HTTPS 301 that breaks naive ngrok http setups.

Pick a stable subdomain you'll reuse forever (e.g. lumion-<your-handle>) and pass it to herd share so Twilio Console webhooks + the TWILIO_*_URL env vars stay valid across tunnel restarts. The --subdomain= flag requires a paid Expose plan; a free Herd Share session will hand you a random *.sharedwithexpose.com URL instead and you'll have to re-patch every restart.

Run from the project root so Herd shares the matching .test site:

cd /Users/leezilla/conductor/workspaces/miashare-app/tallahassee
herd share --subdomain=<your-subdomain>

Your forwarding URL is https://<your-subdomain>.sharedwithexpose.com. Herd opens a browser tab to the Expose dashboard so you can watch incoming webhook traffic. The tunnel stays up until you Ctrl+C (or close the dashboard tab); restarting it later with the same --subdomain= reattaches the same URL — no env-var or Twilio-Console re-patching needed.

cloudflared and bare ngrok are unsupported in our docs.

9. Vapi Console setup (skip if SMS / Verify only)

Vapi (vapi.ai) is the voice-AI orchestrator that sits between Twilio (carrier) and the conversational LLM (gpt-5.4-mini, Claude, etc). For inbound, Twilio bridges the call leg to Vapi via Media Streams; Vapi POSTs an assistant-request webhook to this app, we mint a per-call MCP bearer token, and reply with a transient assistant config that points the LLM at our /mcp/voice-agent server. For outbound, this app calls Vapi's REST POST /call directly.

You need the DEV Vapi org — never production. If your team has a shared DEV org, accept the invite to it (don't create a new personal org — billing).

Skip this entire step if you're only testing SMS or Verify. The whitelist gate, scenario, and env steps below still apply.

  1. Sign up / log in at dashboard.vapi.ai. Use the org switcher (top-left button labeled with the org name) to confirm you're on the DEV org, never production.
  2. API Key (Private) — Sidebar MANAGE → API Keys (URL dashboard.vapi.ai/org/api-keys) → in the Private API Keys section click Add Key → fill the New Private API Key dialog and Create Private Token. Capture the value (shown once) as VAPI_API_KEY.
    • Name: anything readable, e.g. Lumion Local Dev.
    • Allowed Origins: leave empty. This restricts the key to specific browser Origin headers. Our VapiClient is server-side PHP — there's no Origin header on the wire, so a non-empty allowlist would 401 every call.
    • Allowed Assistants: leave empty ("Leave empty for all assistants"). Lumion never references pre-saved assistant IDs — we build transient configs in BuildVapiAssistantConfig and pass them inline to POST /call. An empty allowlist permits all assistants (and is required for transient ones to work).
    • Transient Assistant: leave ON (default). Every Lumion voice call ships its assistant config inline. Switching this OFF makes POST /call reject the inline assistant payload — you'll get "assistant must be a saved assistant ID" errors at call start.
    • Why Private and not Public: the Public API Keys section ("Client-side SDK access") is for browser SDK initialization. Lumion never uses the browser SDK; all Vapi REST calls (POST /call, POST /phone-number, PATCH /phone-number/{id}, control URL POSTs) happen server-side from VapiClient as Bearer-token auth. A Public key would 401 every one of those.
  3. BYO SIP Trunk credential — Sidebar MANAGE → Integrations (URL dashboard.vapi.ai/settings/integrations) → scroll to the Phone Number Providers section → click the SIP Trunk card → Configure New SIP Trunk → fill the form below → Save SIP Trunk. Capture the credential ID from the resulting list as VAPI_SIP_CREDENTIAL_ID. (REST equivalent: POST https://api.vapi.ai/credential with provider byo-sip-trunk.)

    Why required even though SIP isn't used: Vapi's byo-phone-number schema (used by RegisterTwilioNumberWithVapi to mint the phoneNumberId we pass to bypass calls) requires a credentialId pointing at a BYO trunk credential, even when no SIP traffic flows — the current Media Streams bypass flow connects Twilio → Vapi over WebSocket. So the form just needs to satisfy Vapi's schema validation; the values do not need to be operationally correct. See config/vapi.php lines 46–64 for the full rationale.

    Form fields (only the two * fields are required by the form; everything else can keep defaults):

    Field Value Notes
    Name *Lumion Local Dev (or anything readable)UI label only.
    Gateway #1 → IP Address / Domain *127.0.0.1 (placeholder)Format-validated by Vapi but never dialed in our flow. Use a real Twilio Elastic SIP signaling IP (twilio.com/docs/sip-trunking/ip-addresses) if you'd rather it look legit.
    Gateway #1 → Port5060 (default)SIP default. Leave alone.
    Gateway #1 → Netmask32 (default)Single-host. Leave alone.

  4. HMAC webhook credential — Sidebar MANAGE → Integrations → scroll to the Server Configuration section → click the Server Configuration card → Add Custom Credential → fill the form below → Save. Capture the credential id from the resulting list as VAPI_WEBHOOK_CREDENTIAL_ID and the Secret Key value as VAPI_WEBHOOK_SECRET.

    The form defaults are tuned for Vapi's newer Standard-Webhooks-style framing; Lumion's VapiWebhookSignatureMiddleware (app/Http/Middleware/VapiWebhookSignatureMiddleware.php) verifies the legacy x-vapi-signature over the raw body, so override several defaults:

    Field Value Why

    Message ID Header(leave empty)Not used.

    Secret is Base64OFF (default)Plain string.

    The credential id has to be attached to BOTH the org-level Server URL (next step) AND each per-phone-number server config — voice:dev-sync handles the per-phone-number attachment in step 13.

  5. Org-level Server URL — Sidebar MANAGE → Integrations → in the Settings sub-sidebar that appears, click Org Settings (URL dashboard.vapi.ai/settings/organization) → scroll to the Server URL field under "Organization Settings" → set value to https://<your-subdomain>.sharedwithexpose.com/webhooks/vapi → expand Authorization below it → attach the HMAC credential from step 4 → Save.

    Without the credential attached at the org level, Vapi delivers org-level webhooks (e.g. assistant-request fallbacks) unsigned and our middleware 401s them.

There is no TWILIO_FROM in this app. Outbound From comes from the per-institution TwilioNumber.sms_number row you'll bind in step 16. TWILIO_TWIML_APP_SID is only required if you intend to test browser-initiated voice — leave it blank for SMS / Verify only.

VAPI_WEBHOOK_* is technically optional but not really. Without VAPI_WEBHOOK_CREDENTIAL_ID, Vapi delivers end-of-call webhooks unsigned and our signature middleware 401s them. voice:dev-sync warns when it's missing but does not abort. Set both to keep the call lifecycle observable.

LOCAL_TWILIO_* are read by the CJ scenarios (app/Scenarios/CustomerJourney/Concerns/CreatesJourneyBase.php::ensureLocalTwilioNumberForInstitution) so step 15's scenario:run automatically creates the SMS Channel + TwilioNumber row pointing at your real number. Idempotent: re-runs reuse the existing row and update the SIDs to whatever's currently in .env. Leave them unset if you'd rather bind manually via Filament (step 16).

11. .env — webhook URLs

Replace <tunnel> with your Herd Share forwarding URL from step 8 (e.g. https://<your-subdomain>.sharedwithexpose.com). Because the subdomain is stable, you set this once — it'll keep working across machine reboots and tunnel restarts.

TWILIO_VOICE_URL=<tunnel>
TWILIO_SMS_URL=<tunnel>
TWILIO_STATUS_CALLBACK_URL=<tunnel>/twilio/status-callback
TWILIO_INBOUND_REQUEST_URL=<tunnel>/incoming-sms-messages/twilio

# Vapi MCP server (only required for voice agent)
VAPI_MCP_SERVER_URL=<tunnel>/mcp/voice-agent

VAPI_MCP_SERVER_URL defaults to APP_URL + /mcp/voice-agent if unset, but APP_URL is your local .test host that Vapi can't reach. Setting it explicitly to the tunnel host is what makes inbound MCP tool calls from Vapi land on your machine. voice:dev-sync warns when this drifts from the detected tunnel host.

12. .env — turn off the fakes

Both .env and .env.example ship with AUTH_FAKE_VERIFICATION=true, and SMS_MOCK defaults to true for many local setups (it's how FakeSmsService gets bound). Flip both to false so real Twilio traffic leaves the box: 

AUTH_FAKE_VERIFICATION=false
SMS_MOCK=false

Then:

php -d memory_limit=512M artisan config:clear

If APP_CONFIG_CACHE_PATH is set or you previously ran config:cache, also re-run php -d memory_limit=512M artisan config:cache.

14. Whitelist your phone (the new gate)

Without this step, every real SMS / Verify / Voice call from your local box will throw Twilio sms blocked: recipient ending in … (or the voice equivalent). The magic Twilio test number +15005550006 is auto-allowed by the boot whitelist; everything else needs a row.

  1. Open https://admin.tallahassee.test/admin and log in as a Lumion admin.
  2. Sidebar → Developers → Twilio Allowed Recipients.
  3. Click Quick-add my phone. This action uses the phone column on your user record — if that column is null the action errors / no-ops; in that case use New twilio allowed recipient and enter (415) 555-0123 free-form. The model normalizes to +14155550123 on save.

Cache (twilio.allowed_recipients, 24h TTL) refreshes write-through on save — no manual purge needed.

17. Smoke test SMS

This step verifies the whitelist gate behavior end-to-end, not real carrier delivery. Carrier delivery on this account is a separate concern (see "Carrier delivery caveat" below) and isn't required for the gate verification to pass.

Service-level: tinker hitting the gate directly

The dispatch path resolves the SMS service via the tagged container so the WhitelistGatedSmsService decorator is in the chain. Mirror that pattern (don't app(TwilioSms::class) directly — that bypasses the gate): 

php -d memory_limit=512M artisan tinker
$service = collect(iterator_to_array(app()->tagged(\Lumion\Shared\Contracts\SmsServiceInterface::class)))
    ->sole(fn ($s) => $s->supports(\Lumion\Shared\Enums\SmsProvider::TWILIO));

$result = $service->send(new \Lumion\Shared\DTOs\SendSmsDto(
    to: \Brick\PhoneNumber\PhoneNumber::parse('+1YOURNUMBER'),
    source: 'MGxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx',  // Messaging Service SID from step 7
    text: 'hello from local',
));

dump($result);

Carrier delivery caveat (why your phone may not actually buzz)

A successful tinker run with status: pending from Twilio doesn't mean the SMS reached your phone. US carriers (T-Mobile, Verizon, AT&T) drop A2P (application-to-person) SMS from long-code numbers (+1NXX-NXX-XXXX) that aren't registered with A2P 10DLC at The Campaign Registry. Twilio still accepts your POST /Messages request and mints an SID, but the message status flips to undelivered shortly after with error_code: 30034.

Check live status by SID:

ACCT=$(grep -E '^TWILIO_ACCOUNT_SID=' .env | cut -d= -f2-)
TOKEN=$(grep -E '^TWILIO_AUTH_TOKEN=' .env | cut -d= -f2-)
curl -s -u "$ACCT:$TOKEN" \
    "https://api.twilio.com/2010-04-01/Accounts/$ACCT/Messages/SMxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx.json" \
    | python3 -m json.tool | grep -E '"(status|error_code|error_message)":'

If you see error_code: 30034, that's A2P 10DLC. Three resolutions, in increasing fidelity:

Path What Effort
Smoke-test via Verify (recommended for dev)Step 18 below. +15005550006 magic number → Twilio Verify always returns approved, no carrier path needed. Exercises the same WhitelistGatedPhoneVerification decorator.Zero.
Toll-free numberBuy a +1-8XX number; toll-free verification (Console form, ~1–2 days). Bypasses 10DLC.Console + wait.
A2P 10DLCConsole → Messaging → Regulatory Compliance → register a Brand + Campaign. Approval typically multi-day. Right call for prod, overkill for dev.Multi-day.

For dev-machine smoke testing, treat carrier delivery as a separate Lumion org-level concern — not a per-dev setup step. The gate behavior (above) is what this step is actually verifying.

Trigger an OTP via any flow that uses TwilioSmsVerification (e.g. portal magic-link login, or any feature gated by SMS verification).

Re-run the orchestrator if anything changed since step 13:

php -d memory_limit=512M artisan voice:dev-sync \
    --tunnel=https://<your-subdomain>.sharedwithexpose.com \
    --institution=lumion-cj-scenarios \

Then either:

  • Inbound — dial your purchased Twilio number from any phone. Twilio bridges to Vapi, Vapi POSTs assistant-request to <tunnel>/webhooks/vapi, this app mints a per-call MCP token and replies with the assistant config, the conversational LLM uses MCP tools at <tunnel>/mcp/voice-agent. 20. Re-runs and cleanup

Tunnel restart (with the same --subdomain):

Nothing to change in .env or Twilio Console — the URL is the same. Just bring the tunnel back up: 

cd /Users/leezilla/conductor/workspaces/miashare-app/tallahassee
herd share --subdomain=<your-subdomain>

You changed the subdomain (or migrated to/from a free tunnel):

# 1. Edit .env: update every TWILIO_*_URL and VAPI_MCP_SERVER_URL to the new Herd Share host
php -d memory_limit=512M artisan config:clear
php -d memory_limit=512M artisan voice:dev-sync --tunnel=https://<new-subdomain>.sharedwithexpose.com --i-confirm-dev-vapi

The voice:dev-sync step also re-patches Vapi's per-phone-number Server URL. If the org-level Vapi Server URL is wrong, fix it in the Vapi Console — voice:dev-sync does not touch org-level settings.

You removed a number from the whitelist UI but the cache feels stale:

php -d memory_limit=512M artisan tinker --execute 'Cache::forget("twilio.allowed_recipients");'

Inspect the live whitelist cache:

php -d memory_limit=512M artisan tinker --execute 'dump(Cache::get("twilio.allowed_recipients"));'

The boot whitelist (magic numbers + DB rows merged) sits in a singleton — if the in-process state seems wrong, restart the worker / dev server in addition to clearing the cache.
