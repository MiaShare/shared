Twilio staging setup

End-to-end checklist that takes Lumion's staging environment (DigitalOcean Ubuntu droplet, Envoyer-deployed) to: "I can place a real call, send a real SMS, and pass a real Verify OTP from staging code, targeting https://lumion-cj-scenarios.cortex.staging.lumion.ai."

Steps

State persists in localStorage on this domain. Separate from the local-setup checklist.

Prerequisite: you've completed the local-setup guide. Staging reuses the same Twilio sub-account, the same Vapi org, and the same purchased phone number. Staging just takes over the webhook URLs from local. Step 15 covers switching back.

The whitelist gate from lhilton/twilio-phone-whitelist applies to staging too (any non-production env), so step 7 — adding your phone — is non-optional for outbound from staging code.

1. Prereqs

  • Envoyer access to the staging project for cortex.staging.lumion.ai. You'll edit env vars through the Environment tab.
  • SSH access to the staging droplet for php artisan commands. Confirm: ssh <user>@<host> 'echo ok'.
  • Branch deployed. Staging must be running a branch that includes the scenario auto-bind helper (ensureLocalTwilioNumberForInstitution in CreatesJourneyBase) added by lhilton/twilio-phone-whitelist. Without it, step 9 falls back to manual UI binding.
  • Horizon supervised on staging. Most setups run Horizon under Supervisor or systemd, restarted on deploy via Envoyer's hooks. Confirm with sudo supervisorctl status horizon (or equivalent).
  • Local guide complete. You need every SID + secret from twilio-local-setup steps 2–9.
Substitute throughout this guide: <user>@<host> = your droplet SSH credentials, <deploy-path> = the Envoyer project root on the droplet (typically /home/envoyer/<project> — check Envoyer → Project → Settings → Project Path).

7. Whitelist your phone (the staging gate)

Without this, every real outbound SMS / Verify / Voice from staging code throws Twilio sms blocked: recipient ending in …. (Inbound to staging is unaffected — the gate is outbound-only.) The magic Twilio test number +15005550006 is auto-allowed by the boot whitelist; everything else needs a row.

  1. Open https://admin.cortex.staging.lumion.ai/admin and log in as a Lumion staff user.
  2. Sidebar → Developers → Twilio Allowed Recipients.
  3. Click Quick-add my phone (uses the phone column on your user record). If that column is null, fall back to New twilio allowed recipient with the free-form input — (415) 555-0123 normalizes to +14155550123 on save.

Cache (twilio.allowed_recipients, 24h TTL) refreshes write-through on save — no manual purge.

10. Configure voice on lumion-cj-scenarios (skip if SMS / Verify only)

Three things need to happen for voice to work, none of which step 8's CJ scenario does:

  1. The institution's organization needs the FeatureSlugEnum::VOICE feature flag enabled (otherwise Gate::allows('manage-voice-features') denies, the Phone-AI settings UI is hidden, and inbound voice compliance rejects the AI bridge → call drops to "no one is available").
  2. A VOICE Channel must exist on the institution.
  3. A TwilioNumber with is_primary=true, ai_enabled=true, your E.164 + MG/PN SIDs, attached to that VOICE Channel — plus a paired VoiceSetting row (without it, inbound voice routing short-circuits with "this number is not yet configured for voice routing").

One copy/paste command does all three. Replace +1YOURNUMBER with your real E.164 from local guide step 3:

cd <deploy-path>/current && \
TARGET_NUM='+1YOURNUMBER' && \
LOCAL_MG=$(grep -E '^LOCAL_TWILIO_MESSAGING_SERVICE_SID=' .env | cut -d= -f2-) && \
LOCAL_PN=$(grep -E '^LOCAL_TWILIO_PHONE_NUMBER_SID=' .env | cut -d= -f2-) && \
echo "Wiring $TARGET_NUM (MG=$LOCAL_MG, PN=$LOCAL_PN) for voice on lumion-cj-scenarios" && \
php artisan tinker --execute "
use App\Feature;
use App\Institution;
use App\Enums\FeatureSlugEnum;
use App\Enums\ChannelType;
use App\Domains\SaaS\Support\OrganizationTierEntitlements;
use App\Models\Communications\Channel;
use App\Models\Communications\TwilioNumber;
use App\Models\Communications\Voice\VoiceSetting;

\$institution = Institution::where('subdomain', 'lumion-cj-scenarios')->firstOrFail();
\$org = \$institution->organization;
if (!\$org) { throw new RuntimeException('Institution #' . \$institution->getKey() . ' has no organization'); }

// 1. Enable VOICE feature override on the org
Feature::firstOrCreate(['slug' => FeatureSlugEnum::VOICE->value], ['name' => 'Voice']);
app(OrganizationTierEntitlements::class)->enableFeatureOverride(\$org, FeatureSlugEnum::VOICE, 'Manual enable for staging voice testing');

// 2. Find/create VOICE channel on the institution
\$voiceChannel = Channel::query()
    ->where('institution_id', \$institution->getKey())
    ->where('type', ChannelType::VOICE)
    ->first()
    ?? Channel::factory()->voiceType()->forInstitution(\$institution)->create();

// 3. Find/create TwilioNumber on that VOICE channel + paired VoiceSetting
\$attrs = [
    'sms_number'             => '$TARGET_NUM',
    '_messaging_service_sid' => '$LOCAL_MG',
    '_phone_number_sid'      => '$LOCAL_PN',
    'is_primary'             => true,
    'ai_enabled'             => true,
];
\$tn = TwilioNumber::query()->where('channel_id', \$voiceChannel->getKey())->first();
if (\$tn === null) {
    \$tn = TwilioNumber::factory()->forChannel(\$voiceChannel)->create(\$attrs);
} else {
    \$tn->update(\$attrs);
}
VoiceSetting::updateOrCreate(
    ['twilio_number_id' => \$tn->getKey()],
    ['timezone' => 'America/New_York', 'ring_timeout_seconds' => 30, 'recording_enabled' => true, 'transcription_enabled' => true]
);

\$ent = app(OrganizationTierEntitlements::class);
echo 'institution      : ' . \$institution->name . ' (#' . \$institution->getKey() . ')' . PHP_EOL;
echo 'voice channel id : ' . \$voiceChannel->getKey() . PHP_EOL;
echo 'twilio number id : ' . \$tn->getKey() . PHP_EOL;
echo 'hasFeature(VOICE): ' . (\$ent->hasFeature(\$org, FeatureSlugEnum::VOICE) ? 'true' : 'false') . PHP_EOL;
dump(\$tn->fresh()->only(['id','channel_id','sms_number','is_primary','ai_enabled','_messaging_service_sid','_phone_number_sid']));
"

After this runs, refresh https://lumion-cj-scenarios.cortex.staging.lumion.ai/settings/phone-numbers (hard refresh / log out & back in if you've been logged in across the change). The Voice/Phone-AI tabs and the Voice Users sidebar entry should now be visible and editable.

To turn voice off later: php artisan tinker --execute "app(\App\Domains\SaaS\Support\OrganizationTierEntitlements::class)->clearFeatureOverrides(\App\Institution::where('subdomain','lumion-cj-scenarios')->firstOrFail()->organization, [\App\Enums\FeatureSlugEnum::VOICE]);"

11. Re-point Twilio + Vapi webhooks at staging

This step disconnects local. After it, every inbound SMS, every inbound voice call, every Vapi webhook lands on staging. Local devs who want to test will need to reclaim — see step 15. 
ssh <user>@<host>
cd <deploy-path>/current
php artisan voice:dev-sync \
    --tunnel=https://admin.cortex.staging.lumion.ai \
    --institution=lumion-cj-scenarios \
    --i-confirm-dev-vapi --bypass-all-safety-checks

Despite the command name, voice:dev-sync isn't tunnel-specific — --tunnel= is just "the public URL where webhooks should be sent." Pointing it at staging patches:

  • Twilio Phone Number → Voice URL + Voice Status Callback + Recording Status Callback
  • Vapi per-phone-number server config (URL + HMAC credential attachment)
  • The institution's TwilioNumber row in the DB (sets vapi_phone_number_id, voice_webhook_base_url)

It does not patch Twilio's Messaging URL on the phone number, the Messaging Service inbound webhook, or the Vapi org-level Server URL — those are manual additions below.

Hostname prompt is interactive. With --bypass-all-safety-checks, the command pauses with Type the hostname 'admin.cortex.staging.lumion.ai' to confirm bypass:. Laravel Prompts requires a real TTY here — piped stdin (echo … | php artisan …) is treated as empty and aborts. Type the hostname when prompted. (If you really need a single-paste command, wrap in a python PTY — see "All-in-one variant" below.)

Without --i-confirm-dev-vapi, the command also pauses with an interactive "Are you using the Vapi DEV org?" prompt. Same as local — answer yes if so.

Vapi unique-constraint collision (recovery)

If voice:dev-sync fails with:

SQLSTATE[23000]: Integrity constraint violation: 1062 Duplicate entry '...' for key 'twilio_numbers.twilio_numbers_vapi_phone_number_id_unique'

Vapi already had your E.164 registered (likely from prior local-dev work) and returned the existing UUID, but another TwilioNumber row in your DB already holds it. Clear it on every other row, then re-run dev-sync: 

php artisan tinker --execute "
use App\Models\Communications\TwilioNumber;
\$voice = TwilioNumber::query()
    ->whereHas('channel', fn(\$q) => \$q->where('type', \App\Enums\ChannelType::VOICE)
        ->whereHas('institution', fn(\$i) => \$i->where('subdomain', 'lumion-cj-scenarios')))
    ->first();
\$updated = TwilioNumber::where('id', '!=', \$voice?->getKey() ?? 0)
    ->whereNotNull('vapi_phone_number_id')
    ->update(['vapi_phone_number_id' => null]);
echo 'Cleared vapi_phone_number_id on ' . \$updated . ' other row(s)' . PHP_EOL;
"

All-in-one variant (auto-confirms hostname via Python PTY)

If you want voice:dev-sync fully automated, wrap it in a Python pseudo-terminal so Laravel Prompts sees a TTY and the auto-typed answer satisfies the hostname check:

python3 - <<'PY'
import os, pty, sys, time
ANSWER = b"admin.cortex.staging.lumion.ai\n"
master, slave = pty.openpty()
pid = os.fork()
if pid == 0:
    os.close(master)
    os.dup2(slave, 0); os.dup2(slave, 1); os.dup2(slave, 2)
    os.close(slave)
    os.execvp("php", ["php", "artisan", "voice:dev-sync",
        "--tunnel=https://admin.cortex.staging.lumion.ai",
        "--institution=lumion-cj-scenarios",
        "--i-confirm-dev-vapi", "--bypass-all-safety-checks"])
os.close(slave)
buf, sent = b"", False
while True:
    try:
        data = os.read(master, 4096)
    except OSError:
        break
    if not data:
        break
    sys.stdout.buffer.write(data); sys.stdout.flush()
    buf += data
    if not sent and b"Type the hostname" in buf:
        time.sleep(0.5)
        os.write(master, ANSWER)
        sent = True
os.waitpid(pid, 0)
PY

python3 is on every Ubuntu base install. The PTY makes stream_isatty(STDIN) return true so Laravel Prompts reads the auto-typed hostname.

Carrier delivery caveat

US carriers (T-Mobile, Verizon, AT&T) drop A2P SMS from long-code numbers without A2P 10DLC registration. Twilio still mints an SID and reports status: pending, then flips to undelivered with error_code: 30034 shortly after. This is org-level and not specific to staging vs local. See local guide step 17 ("Carrier delivery caveat") for the curl recipe to inspect message status by SID and the three resolution paths. For dev smoke testing, use Verify (step 13) instead — +15005550006 always returns approved.

Common failure: dial answers with "we are sorry, no one is available to take your call"

That's the fallback in InitialStep::handleDirectRouting — your call reached staging, but the AI bridge never fired and there's no ring group with members on the resolved TwilioNumber. The usual culprits, in order:

  1. Two TwilioNumber rows for the same E.164, one on a stale institution (typically voice-demo-ai-inbound from voice:ai_agent_answers_inbound). The inbound resolver does TwilioNumber::where('sms_number', $to)->whereResolvedVoiceChannelIsVoice()->first() — no orderBy, no is_primary filter — so it picks the older one, which is the wrong institution. Fix: soft-delete the duplicate. 
    php artisan tinker --execute "
use App\Institution;
use App\Models\Communications\TwilioNumber;
\$ghost = Institution::where('subdomain', 'voice-demo-ai-inbound')->first();
if (\$ghost) {
    foreach (TwilioNumber::whereHas('channel', fn(\$q) => \$q->where('institution_id', \$ghost->getKey()))->get() as \$v) {
        echo 'Soft-deleting TwilioNumber #' . \$v->getKey() . PHP_EOL;
        \$v->delete();
    }
}
\$picked = TwilioNumber::where('sms_number', '+1YOURNUMBER')->whereResolvedVoiceChannelIsVoice()->first();
echo 'Resolver now picks: ' . (\$picked ? '#' . \$picked->getKey() . ' on ' . \$picked->channel?->institution?->subdomain : 'NONE') . PHP_EOL;
"
  2. VOICE feature flag not enabled on the org → voice compliance throws → AI bridge skipped → falls through to direct routing → no ring group → fallback. Re-run step 10's tinker block.
  3. ai_enabled=false on the resolved TwilioNumber → AI bridge skipped. Re-run step 10's tinker block.

If the call connects but goes silent after the greeting, check that VAPI_MCP_SERVER_URL in Envoyer matches https://admin.cortex.staging.lumion.ai. If /voice/twiml returns 502 in Twilio's debugger, see app/Console/Commands/Communications/VoiceDevSyncCommand.md "Common failure modes". If Vapi end-of-call webhooks 401, the HMAC credential isn't attached to the per-phone-number server config — re-run voice:dev-sync.

15. Switching back to local

Twilio + Vapi resources have one webhook destination at a time. While staging owns the webhooks, your local tunnel won't receive any inbound traffic. To reclaim for local development: 

# On your local machine, with the tunnel running
herd share --subdomain=<your-local-subdomain>

# In another terminal
cd /Users/leezilla/conductor/workspaces/miashare-app/tallahassee
php -d memory_limit=512M artisan voice:dev-sync \
    --tunnel=https://<your-local-subdomain>.sharedwithexpose.com \
    --institution=lumion-cj-scenarios \
    --i-confirm-dev-vapi

This re-points the same Twilio Phone Number + Vapi per-phone-number config back at your local tunnel. Staging stops receiving webhooks until someone re-runs the staging voice:dev-sync from step 11.
